# 途正英语AI分级测评 — 后端接口需求清单

> 编写：火鹰科技 | 日期：2026-03-29

---

## 一、当前问题总结

小程序前端测评流程已开发完成，但实际测试中发现**三个核心问题全部指向后端接口**：

1. **听不到外教提问**：`/api/v1/test/start` 和 `/api/v1/test/evaluate` 返回的 `question.audioUrl` 为空或不可用，导致前端没有音频可播放。
2. **录音提交后总是0分**：`/api/v1/test/evaluate` 接口在收到 `audioUrl`（用户录音OSS地址）后，未能正确识别用户语音内容进行评分，始终返回0分。
3. **题号不从第1题开始**：`/api/v1/test/start` 接口返回的 `totalAnswered` 不为0，说明后端没有为新测评创建全新的session，而是复用了之前未完成的session。

---

## 二、需要后端确认和修复的接口

### 接口1：`POST /api/v1/test/start` — 创建测评会话

**当前问题：**

前端调用此接口后，返回的 `question.audioUrl` 字段是否有值？如果为空，前端就无法播放外教提问语音。另外 `totalAnswered` 字段在新测评时应该返回0，但实际返回了非0值。

**后端需要确认的事项：**

| 序号 | 确认项 | 说明 |
|------|--------|------|
| 1 | `question.audioUrl` 是否有值 | 每道题必须返回一个可直接访问的音频URL（mp3/m4a格式），前端用 `InnerAudioContext` 播放 |
| 2 | audioUrl 是否可公网访问 | 小程序的 `InnerAudioContext` 需要直接访问URL，不能有鉴权拦截，建议用CDN或OSS公开链接 |
| 3 | 新测评的 `totalAnswered` | 新创建的session，`totalAnswered` 必须返回0，不应复用之前未完成的session |
| 4 | 是否支持强制创建新session | 建议增加 `forceNew: true` 参数，前端传入时强制创建新session，不复用旧的 |

**期望的返回格式（已约定）：**

```json
{
  "code": 200,
  "data": {
    "sessionId": "sess_xxx",
    "currentSubLevel": "PRE1",
    "currentMajorLevel": 0,
    "questionIndex": 1,
    "totalAnswered": 0,
    "question": {
      "questionId": "q_001",
      "audioUrl": "https://cdn.xxx.com/audio/pre1_q1.mp3",
      "questionText": "What is your name?",
      "subLevel": "PRE1"
    }
  }
}
```

**关键要求：`question.audioUrl` 必须是一个有效的、可公网直接访问的音频文件URL。** 这是整个测评流程的起点，没有音频就无法进行听力测试。

---

### 接口2：`POST /api/v1/test/evaluate` — 提交回答并评分

**当前问题：**

前端提交了 `audioUrl`（用户录音的OSS地址）和 `recognizedText`（前端语音识别的文本），但后端始终返回0分，evaluation.feedback 显示"没有回答"。

**后端需要确认的事项：**

| 序号 | 确认项 | 说明 |
|------|--------|------|
| 1 | 后端是否自己做语音识别 | 如果后端依赖前端传的 `recognizedText`，那当前端识别失败时就会传空字符串，后端就判定为"没回答"。**建议后端自己根据 `audioUrl` 做Whisper语音识别**，不依赖前端 |
| 2 | `audioUrl` 是否能被后端访问 | 前端上传录音到 `/api/v1/test/upload-audio` 后拿到的URL，后端evaluate时能否访问到这个文件 |
| 3 | 评分逻辑是否正常 | 即使 `recognizedText` 为空，只要 `audioUrl` 有值，后端应该自己转写后评分，而不是直接返回0分 |
| 4 | `evaluation.passed` 的判定标准 | 多少分算通过？前端需要这个来决定是否升级 |

**前端提交的参数格式：**

```json
{
  "sessionId": "sess_xxx",
  "questionId": "q_001",
  "audioUrl": "https://oss.xxx.com/recordings/sess_xxx_q001.mp3",
  "recognizedText": "My name is Tom",
  "duration": 5000
}
```

**期望的返回格式（status=continue）：**

```json
{
  "code": 200,
  "data": {
    "evaluation": {
      "passed": true,
      "score": 85,
      "feedback": "Good pronunciation! Your answer is clear and accurate.",
      "scoreDetail": { "pronunciation": 80, "grammar": 90, "vocabulary": 85 }
    },
    "status": "continue",
    "currentSubLevel": "PRE1",
    "currentMajorLevel": 0,
    "questionIndex": 2,
    "totalAnswered": 1,
    "question": {
      "questionId": "q_002",
      "audioUrl": "https://cdn.xxx.com/audio/pre1_q2.mp3",
      "questionText": "How old are you?",
      "subLevel": "PRE1"
    }
  }
}
```

**最关键的一点：后端必须具备根据 `audioUrl` 自主识别用户语音的能力。** 前端的同声传译插件（微信WechatSI）对英语识别准确率很低，不能作为评分依据。后端应该用Whisper或其他ASR服务自己转写录音，然后再用LLM评分。

---

### 接口3：`POST /api/v1/test/upload-audio` — 上传录音

**当前问题：** 需要确认此接口是否正常工作。

| 序号 | 确认项 | 说明 |
|------|--------|------|
| 1 | 上传后返回的URL是否可访问 | 返回的 `audioUrl` 必须是后端evaluate接口能访问到的有效URL |
| 2 | 支持的音频格式 | 前端录音格式为 mp3，16kHz，单声道 |
| 3 | 文件大小限制 | 前端录音最长60秒，mp3格式大约700KB以内 |

**前端上传方式：** `wx.uploadFile`，字段名 `file`，附带 `sessionId` 和 `questionId` 作为formData。

**期望返回：**

```json
{
  "code": 200,
  "data": {
    "audioUrl": "https://oss.xxx.com/recordings/sess_xxx_q001.mp3"
  }
}
```

---

### 接口4（可选但强烈建议）：`POST /api/v1/test/tts` — 文本转语音

**用途：** 当题目没有预录音频（`audioUrl`为空）时，前端调用此接口将题目文本转为语音播放给用户。

| 序号 | 说明 |
|------|------|
| 1 | 如果每道题都有预录好的外教音频，此接口可以不实现 |
| 2 | 如果题库中有部分题目没有预录音频，则必须实现此接口作为兜底 |
| 3 | 建议用Azure TTS或其他英语TTS服务，返回音频URL |

**请求参数：**

```json
{
  "text": "What is your name?",
  "voice": "en-US-female",
  "speed": 0.85
}
```

**期望返回：**

```json
{
  "code": 200,
  "data": {
    "audioUrl": "https://cdn.xxx.com/tts/generated_xxx.mp3"
  }
}
```

## 三、核心问题优先级

| 优先级 | 问题 | 影响 | 后端动作 |
|--------|------|------|----------|
| **P0 最高** | `question.audioUrl` 为空 | 用户听不到题目，整个测评无法进行 | 确保每道题都有有效的音频URL |
| **P0 最高** | evaluate 不自己做语音识别 | 用户录音后总是0分 | 后端根据audioUrl自主做Whisper转写+LLM评分 |
| **P1 高** | start 接口复用旧session | 题号不从1开始 | 新测评强制创建新session |
| **P2 中** | upload-audio 返回URL不可用 | evaluate拿不到录音文件 | 确认OSS上传和URL可访问性 |
| **P3 低** | TTS接口未实现 | 没有预录音频的题目无法播放 | 如果所有题目都有预录音频则可忽略 |

## 四、前端已完成的工作

前端已经做好了以下所有环节，**只要后端接口数据正确返回，整个流程就能跑通**：

1. 进入题目后自动播放 `question.audioUrl` 中的音频（已修复InnerAudioContext复用bug，每次重建实例）
2. 音频播放完毕后自动显示录音按钮（点击开始/点击结束模式）
3. 录音完成后自动上传到 `/api/v1/test/upload-audio`
4. 上传成功后调用 `/api/v1/test/evaluate` 提交评分
5. 如果前端同声传译识别到文本，会一并传给后端（`recognizedText`字段），但**不应作为评分的唯一依据**
6. 15秒音频播放超时保护（防止音频加载失败导致卡死）
7. 至少答完10题才允许定级结束的前端保底逻辑
8. 结果页的重测+确认最终评级机制

## 六、总结

**一句话概括：后端需要确保每道题都有可播放的音频URL，并且evaluate接口能根据用户录音自主做语音识别和评分，不依赖前端传的recognizedText。** 这两个问题解决了，整个测评流程就能跑通。
